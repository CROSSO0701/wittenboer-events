# Design Context — Wittenboer Events Redesign

## Users

**Primary**: Event organizers, festival directors, corporate event managers, venue owners in the Brabant region and wider NL. They are professionals who book production services (sound, light, stage, power, artist management) for their events.

**Context of use**: They browse during work hours on laptops and phones, comparing vendors. They are vetting Wittenboer against 2-3 other production companies. They need to trust the quality quickly and get to contact details without friction.

**Job-to-be-done**:
1. Confirm this company can handle my event (festival / corporate / tuinfeest / beurs / presentation)
2. See proof of past work (photos, client names, project types)
3. Reach out for a quote or info conversation

## Brand Personality

**Voice**: Professional Dutch B2B using formal "u" form. Warm, capable, unpretentious. Uses event-industry vernacular: *meedenken*, *ontzorgen*, *uit handen nemen*, *van A tot Z*. Not marketing-speak, not corporate jargon.

**Three words**: warm, capable, theatrical

**Emotional target**: Confidence. Visitor should feel "this is clearly a pro, I don't have to worry about whether they can handle my event."

## Aesthetic Direction

**Theme**: Light interface (off-white base) with cinematic dark photo sections as punctuation. Rationale: users browse during the day comparing vendors — light is the correct default. The dark photos show the *product* (stage lighting against night-time events), so the contrast is meaningful.

**Typography**: Darker Grotesque (display + body, single family, weights 300–900) with Sometype Mono for accent details (phone, address, project numbers). Rejected the reflex picks (Inter, Fraunces, IBM Plex, Geist, Satoshi, Cabinet Grotesk) because they create monoculture.

**Color**: 
- Base neutrals tinted toward amber (chroma 0.008, hue 80) — subtle warmth, not obvious
- Single accent: `oklch(70% 0.14 60)` — stage-lamp amber, used sparingly on CTAs and focus states
- Dark photo sections: `oklch(12% 0.008 80)` background, never pure black
- OKLCH throughout, no HSL, no hex for anything dynamic

**Layout**: Asymmetric magazine spreads. Hero uses 60/40 split with typography left, cinematic photo right. Services section uses a 2-column zig-zag, not 3-column cards. Testimonials are inline magazine-quotes, not card carousels.

**Motion**: Gentle stagger reveals on scroll using Motion library. Exponential easing (ease-out-quart). Respects `prefers-reduced-motion`. NO bounce, NO spring-overshoot, NO parallax-distortion.

**References** (taste direction): concert program booklets, stage-documentary photography, film-credits typography layouts. NOT: fashion-magazine, tech-startup, SaaS-dashboard.

**Anti-references**: ~~Linear.app dark-tech dashboards~~, ~~Stripe developer-marketing~~, ~~Vercel developer-tooling~~, ~~fashion-editorial serif-heavy~~. These are all beautiful but wrong for a Brabantse event-productie bedrijf.

## Design Principles

1. **Content first, design invisible**. If someone notices the design before the photos or Marnix's proof-of-work, we failed.
2. **One accent, used rarely**. Amber only on CTAs, focus states, and the occasional pull-quote. Everything else is neutral.
3. **Real copy, real numbers, real people**. No Lorem Ipsum (ironic because the live site has one). Use the actual testimonials from Thomas de Groot, Bert van Kronenburg, Berk Music. Preserve the formal "u" form.
4. **Mobile first means mobile-designed, not mobile-fallback**. Stack cinematically on mobile (full-bleed photos, big asymmetric type), not shrunk-desktop.
5. **No AI slop tells, ever**. Explicit ban list enforced in CSS and component structure (see DESIGN.md).

## Accessibility

- WCAG AA minimum (AAA on critical paths)
- All body text ≥ 4.5:1 contrast
- Visible focus rings (not just browser default)
- `prefers-reduced-motion` respected throughout
- Alt text on every photo describing the event shown
- Touch targets ≥ 44px
- `text-wrap: pretty` and `text-wrap: balance` to avoid orphaned words

## Technical Constraints

- Next.js 16.2.1 (App Router, Turbopack stable, React Compiler)
- React 19
- Tailwind CSS v4.1 (CSS-first config via `@theme`)
- Motion library (`motion/react`, not `framer-motion`)
- Deployment: GitHub → Vercel auto-deploy
- Target: Lighthouse Performance ≥ 95 mobile, Accessibility = 100
- No framework beyond this stack unless justified in code comment
